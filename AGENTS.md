> **always respond in 한글** — 모든 응답·문서·주석은 한글로 작성

## 1. 항상 적용되는 지침

- 실행 가능한 프로덕션 수준 코드 작성
- 모르면 추측하지 말고 **가정 명시 or 질문** (외부 API, 라이브러리 동작 등은 검색 우선)
- "정답"보다 "이 프로젝트에 맞는 선택" 우선
- 기존 코드의 의도와 규칙을 먼저 파악한 뒤 수정한다
- `PROGRESS.md`, `docs/plan/`은 필요 시점에 반드시 다시 확인한다

## 2. 상황에 따라 적용되는 지침

**진행 순서:**

1. `/docs/plan/`에 간결한 계획 작성 **단, 간단한 작업일경우는 제외**
2. 주요 결정사항이 있을경우 사용자에게 확인후 진행
3. 구현 진행

**사용자의 확인이 필요한 경우:**

- 요구사항이 불명확
- 되돌리기 어려운 구조 변경

### 기존 작업을 이어서 할 때만

읽는 순서:

1. `PROGRESS.md` — 현재 상태, 남은 작업
2. `docs/plan/` — 설계 의도

이미 결정된 사항은 구체적 이유 없이 재논의하지 않는다.

## 3. 문서 규칙

> "항상" 만들지 않는다. **필요할 때만 생성/업데이트.**

### PROGRESS.md (프로젝트 루트)

**생성 시점:** 여러 단계 작업, 다음 세션에서 이어질 가능성이 있을 때

**필수 항목:**

- **Current Status** — 다음 세션이 이것만 보고 바로 이어갈 수 있도록
- **TODO / In Progress / Done**
- **Known Issues** — 미해결 버그, 기술 부채, 보류된 결정

## 4. Git 규칙

- 브랜치명은 `feature` / `fix` / `hotfix` / `refactor` / `chore` 접두어 사용
- 커밋 메시지는 Conventional Commits 형식 사용: `<type>(<scope>): <한글 요약>`
